# 專案開發文件

## 目的

這份文件是本專案的唯一入口文件。

每次重新開始開發時，Codex 必須先讀取本文件，再決定是否讀取其他技能文件。
目標是：

1. 降低 token 消耗
2. 降低不必要上下文載入
3. 讓每次開發都有一致流程
4. 讓進度能被持續追蹤

## 強制流程

每次重新進入本專案時，依序執行：

1. 先讀取 [專案開發文件.md](C:/dev/陳情系統/leyicasedemo/專案開發文件.md)
2. 讀取 [進度專案.md](C:/dev/陳情系統/leyicasedemo/進度專案.md) 最後一段與未完成事項
3. 根據當前任務，僅讀取必要的技能文件
4. 開始分析、修改、驗證
5. 完成後，把結果追加記錄到 [進度專案.md](C:/dev/陳情系統/leyicasedemo/進度專案.md)

## 不可違反規則

1. 不要每次都重新讀取所有 md 文件
2. 不要一次載入所有技能文件
3. 只有在任務需要時才讀對應技能文件
4. 修改完成後，必須更新進度文件
5. 新增規則時，優先更新本文件，不要把流程散落在其他地方

## 專案定位

本專案不是單一 SaaS，而是「可快速複製的客戶 Demo 模板」。

核心特性：

1. 每個客戶是獨立系統
2. 可快速換品牌、換內容、換頁面
3. 前台、後台、政績專區要能持續擴充
4. 以低成本交付 Demo 為主

## 當前開發主軸

目前優先順序如下：

1. 模板化設定架構
2. 前台頁面管理機制
3. Notion / n8n 資料底座
4. 政績專區重構
5. 新客戶複製 SOP

## 技能文件讀取規則

### 遇到以下任務時再讀

- 模板化、設定驅動、客戶複製：讀 `技能-模板架構.md`
- 前台頁面新增、父子層、拖曳排序、導覽結構：讀 `技能-前台頁面管理.md`
- Notion schema、欄位規劃、資料模型：讀 `技能-Notion資料庫.md`
- webhook、workflow JSON、n8n 節點設計：讀 `技能-n8n工作流.md`
- 政績專區資訊架構、視覺層級、素材規劃：讀 `技能-政績專區.md`
- GitHub、提交、交付流程：讀 `技能-GitHub協作.md`

## 修改原則

1. 優先做小步驟、可驗證修改
2. 優先維持相容性，不先做破壞性重構
3. 先定資料模型，再做 UI
4. 先定規格，再串接外部服務
5. 文件與程式同步更新

## 完成定義

一個任務要算完成，至少要滿足：

1. 程式已修改
2. 相關文件已更新
3. 已在 `進度專案.md` 留下記錄
4. 已列出尚未完成的風險或下一步
